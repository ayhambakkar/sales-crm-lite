# Sales CRM Lite — Security Concept

## Security Philosophy

Security is not a feature added at the end — it is a constraint built into every layer. This document defines the security model for Sales CRM Lite and maps each threat to a concrete mitigation. All mitigations are implemented at the application level (no framework, no magic).

The threat model assumes:
- The application runs on a shared or VPS hosting environment
- Users are trusted employees, not anonymous internet users
- The greatest risks are: unauthorized data access, credential theft, and client-side injection

---

## 1. Authentication

### Mechanism

Session-based authentication using PHP's native session handler. No JWT, no cookie-based tokens.

### Login Flow

```
1. User submits POST /login (email + password)
2. CsrfMiddleware validates CSRF token → abort 403 if invalid
3. AuthController::login()
   a. Sanitize email input (trim, lowercase)
   b. Fetch user row: SELECT * FROM users WHERE email = :email LIMIT 1
   c. If no row found → generic error ("Invalid credentials") — do NOT reveal which field is wrong
   d. If user.is_active = 0 → "Account is disabled" — intentionally distinct message
   e. password_verify($input, $user['password_hash'])
      → false: generic error + increment fail counter (future: lockout)
      → true: proceed
   f. session_regenerate_id(true)  ← CRITICAL: prevents session fixation
   g. Store in session: user_id, role, last_activity = time()
   h. Redirect to /
4. Any exception → log internally, show generic error to user
```

### Password Storage

```php
// Creating:  cost 12 is the minimum recommended for 2024+
$hash = password_hash($plaintext, PASSWORD_BCRYPT, ['cost' => 12]);

// Verifying:
$ok = password_verify($plaintext, $hash);

// Future-proofing:
if (password_needs_rehash($hash, PASSWORD_BCRYPT, ['cost' => 12])) {
    // rehash and update DB on next login
}
```

| Property | Value |
|---|---|
| Algorithm | bcrypt |
| Cost factor | 12 |
| Hash length | 60 chars (stored as VARCHAR(255) for upgrade headroom) |
| Salt | Auto-generated by `password_hash()` per call |

**No plain-text passwords are ever stored, logged, or transmitted.**

## 2. Session Management

### Session Configuration

Applied in `public/index.php` before `session_start()`:

| `php.ini` / runtime directive | Value | Reason |
|---|---|---|
| `session.cookie_httponly` | `true` | Blocks JavaScript from reading the session cookie |
| `session.cookie_secure` | `true` (production) | Cookie only sent over HTTPS |
| `session.cookie_samesite` | `Strict` | Prevents CSRF via cross-site cookie submission |
| `session.use_strict_mode` | `true` | Rejects uninitialized session IDs |
| `session.use_only_cookies` | `true` | No session ID in URL (`PHPSESSID=...` in query string) |
| `session.gc_maxlifetime` | `7200` | 2-hour server-side garbage collection window |

### Session Lifetime Enforcement

Inactivity timeout is enforced at the **application layer** (not solely by GC, which is probabilistic):

```
On every authenticated request:
  if (time() - session.last_activity) > SESSION_LIFETIME (7200s):
      session_destroy()
      redirect /login with flash "Session expired"
  else:
      session.last_activity = time()   ← touch on activity
```

### Session Fixation Prevention

`session_regenerate_id(true)` is called on every successful login. The old session ID is immediately invalidated (`true` = delete old session file).

### Session Invalidation on Logout

```
POST /logout
  session_unset()
  session_destroy()
  setcookie(session_name(), '', time() - 3600, '/')  ← expire cookie client-side
  redirect /login
```

## 3. CSRF Protection

All state-changing requests (POST, which covers create / update / delete / logout) require a valid CSRF token.

### Token Generation

A token is generated once per session and stored server-side:

```
On GET request (form render):
  if session.csrf_token is empty:
      session.csrf_token = bin2hex(random_bytes(32))   ← 256-bit random
  embed token in every form as hidden input:
      <input type="hidden" name="csrf_token" value="<?= session.csrf_token ?>">
```

### Token Validation

```
On POST request:
  CsrfMiddleware::handle()
    expected = session.csrf_token
    received = $_POST['csrf_token'] ?? ''
    if (!hash_equals($expected, $received)):
        abort 403 "Invalid request"
```

`hash_equals()` is used instead of `===` to prevent timing attacks.

### Token Rotation

The CSRF token is **regenerated after each validated POST** that modifies data. This limits the window of CSRF token reuse.

## 4. Authorization — Role-Based Access Control

### Role Model

| Role | Capabilities |
|---|---|
| `admin` | Full access to all records; user management; all reports |
| `sales_rep` | CRUD on own leads, customers, follow-ups; own dashboard data only |

### Enforcement Points

Authorization is checked at **two layers** — defense in depth:

**Layer 1: Route-level (Router)**
- Routes tagged with `admin` middleware are inaccessible to `sales_rep` users
- Example: `/users` (user management) aborts 403 for non-admins

**Layer 2: Controller-level (record ownership)**
- Before any read/write on a Lead, Customer, or Follow-Up:
  ```
  if (Auth::isAdmin() === false && $record['assigned_user_id'] !== Auth::user()['id']):
      abort 403
  ```
- This prevents Sales Rep A from accessing Sales Rep B's records by manipulating the URL

### IDOR Prevention

Insecure Direct Object Reference is the most common CRM vulnerability. Every model method that fetches a single record by ID includes a user-scope check:

```sql
-- LeadModel::findByIdScoped($id, $userId, $isAdmin)
SELECT * FROM leads
WHERE id = :id
  AND (:is_admin = 1 OR assigned_user_id = :user_id)
LIMIT 1;
```

If the record does not exist **or** the user is not authorized, the result is `null` — indistinguishable to the caller. The controller then calls `abort(404)`, not `abort(403)`. This prevents enumeration of other users' record IDs.

## 5. SQL Injection Prevention

**All database interaction uses PDO with prepared statements.** No string concatenation in SQL ever.

| Practice | Status |
|---|---|
| Prepared statements for all queries | Required |
| `PDO::ATTR_EMULATE_PREPARES = false` | Real DB-level prepared statements |
| Dynamic ORDER BY / column names | Whitelisted — never interpolated from user input |
| LIMIT / OFFSET | Cast to int before use: `(int)$_GET['page']` |

**Safe pattern for dynamic sort columns:**
```php
// Whitelist allowed columns — never trust user input directly
$allowed = ['created_at', 'company', 'status'];
$sort = in_array($_GET['sort'] ?? '', $allowed) ? $_GET['sort'] : 'created_at';
// $sort is now safe to interpolate — it came from our whitelist, not user input
$sql = "SELECT * FROM leads ORDER BY {$sort} DESC";
```

## 6. XSS Prevention

All user-supplied data is escaped on **output**, not on input. Input is stored raw (preserving integrity); output is always escaped in views.

### Output Escaping Rule

Every PHP variable echoed in a view must pass through the escape helper:

```php
// Helper function in src/Core/helpers.php
function e(mixed $value): string {
    return htmlspecialchars((string)$value, ENT_QUOTES | ENT_SUBSTITUTE, 'UTF-8');
}
```

```html
<!-- In every view template — no exceptions -->
<td><?= e($lead['company']) ?></td>
<td><?= e($lead['notes']) ?></td>
```

### Content Security Policy (HTTP Header)

Applied via `header()` in `public/index.php`:

```
Content-Security-Policy:
  default-src 'self';
  script-src  'self' https://cdn.tailwindcss.com;
  style-src   'self' 'unsafe-inline' https://cdn.tailwindcss.com;
  img-src     'self' data:;
  font-src    'self';
  frame-ancestors 'none';
```

This blocks inline script injection even if an XSS vulnerability were introduced.

## 7. Security HTTP Headers

Set in `public/index.php` for every response:

| Header | Value | Effect |
|---|---|---|
| `X-Frame-Options` | `DENY` | Blocks clickjacking via iframe embedding |
| `X-Content-Type-Options` | `nosniff` | Prevents MIME-type sniffing |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Limits referrer data leakage |
| `X-XSS-Protection` | `1; mode=block` | Legacy XSS filter for older browsers |
| `Strict-Transport-Security` | `max-age=31536000` (production) | Forces HTTPS for 1 year |
| `Content-Security-Policy` | (see above) | Script/style injection prevention |

## 8. Input Validation

Validation is always **server-side**. Client-side validation is a UX convenience only — it is never trusted.

### Validation Rules (examples)

| Field | Rule |
|---|---|
| `email` | `filter_var($v, FILTER_VALIDATE_EMAIL)` |
| `first_name`, `last_name` | Non-empty, max 100 chars, strip tags |
| `status` | Must be in the known ENUM set (whitelist) |
| `due_date` | Valid date via `DateTime::createFromFormat('Y-m-d', $v)` |
| `assigned_user_id` | Must exist in `users` table AND be active |
| `id` parameters in URL | Cast to `(int)` immediately; reject if 0 |

### Sanitization vs Validation

- **Sanitize** on input: trim whitespace, cast types (`(int)`, `(bool)`)
- **Validate** after sanitize: check constraints; collect all errors before returning
- **Escape** on output: `e()` helper in every view
- **Never** run `strip_tags()` or `htmlspecialchars()` on data before storing — it corrupts the original value and complicates future use

## 9. Error Handling & Information Disclosure

| Environment | Exception handling |
|---|---|
| Development (`APP_ENV=development`) | Full exception message + stack trace shown in browser |
| Production (`APP_ENV=production`) | Generic "Something went wrong" page; exception logged to `storage/logs/error.log` |

Error log format:
```
[2025-11-14 10:22:01] ERROR: PDOException: SQLSTATE[23000] ... in Model.php:42
Stack trace: #0 LeadModel::findById() ...
```

**Sensitive information never exposed:**
- No stack traces in production
- No SQL queries in error output
- No user data in error messages
- DB credentials only in `.env` (git-ignored)

## 11. Security Checklist (Pre-Deploy)

Use this checklist before merging any feature to `main`:

### Authentication & Session
- [ ] Passwords hashed with bcrypt cost ≥ 12
- [ ] `session_regenerate_id(true)` called on login
- [ ] Session cookie flags: HttpOnly, Secure (prod), SameSite=Strict
- [ ] Session expiry enforced at application layer (not only GC)
- [ ] Logout destroys session and expires cookie

### Authorization
- [ ] Every controller method checks `Auth::check()` (via middleware)
- [ ] Every single-record fetch includes user-scope WHERE clause
- [ ] Admin-only routes return 403 (not 404) to authenticated non-admins
- [ ] IDOR check: fetching a record by ID you don't own returns 404

### Input & Output
- [ ] Every `$_POST` / `$_GET` value is validated server-side before use
- [ ] Every echoed variable uses `e()` helper
- [ ] No raw `echo $_GET['x']` anywhere in views

### Database
- [ ] Zero raw string interpolation in SQL
- [ ] `PDO::ATTR_EMULATE_PREPARES = false`
- [ ] Dynamic sort columns validated against whitelist

### CSRF
- [ ] Every POST form includes `<input type="hidden" name="csrf_token">`
- [ ] `CsrfMiddleware` runs on all POST routes
- [ ] `hash_equals()` used for token comparison

### Headers & Config
- [ ] All security headers set in `index.php`
- [ ] `APP_ENV=production` in live `.env`
- [ ] `.env` is in `.gitignore` and not committed
- [ ] `display_errors = Off` in production `php.ini`
- [ ] Error logging enabled and log directory is not web-accessible

## Threat Summary

| Threat | OWASP Category | Mitigation |
|---|---|---|
| Password cracking | A02 Cryptographic Failures | bcrypt cost 12; no plain storage |
| Session hijacking | A07 Identification Failures | HttpOnly + Secure cookies; SameSite=Strict |
| Session fixation | A07 Identification Failures | `session_regenerate_id(true)` on login |
| CSRF | A01 Broken Access Control | CSRF tokens with `hash_equals()` validation |
| IDOR | A01 Broken Access Control | Scoped SQL queries; 404 on unauthorized access |
| SQL Injection | A03 Injection | PDO prepared statements; no string interpolation |
| XSS (Stored) | A03 Injection | `htmlspecialchars()` on all output + CSP header |
| Broken Auth | A07 Identification Failures | Bcrypt; session expiry; inactive user check |
| Sensitive data exposure | A02 Cryptographic Failures | `.env` git-ignored; no credentials in code |
| Clickjacking | A05 Security Misconfiguration | `X-Frame-Options: DENY` |
| Info disclosure | A05 Security Misconfiguration | Generic error pages in production |
